# SPEC Appendix — Objections to Option D + Typed Graphs

This appendix documents objections, challenges, and arguments AGAINST
the proposed direction (keep the `GraphBackend` protocol per issue #4
Option D, AND adopt ontologically typed named graphs per `SPEC.md`
§ R9.1). Each objection is steelmanned first, then given a preliminary
counter.

The purpose of this document is not to win the argument. It is to
ensure the costs and tradeoffs are visible so that when the design
hits friction in practice, the friction is recognized for what it is
rather than treated as a surprise.

---

## 1. Standards-compliance objections

### 1.1 "You are reinventing what the W3C already specified."

**The argument:** RDF 1.1 datasets, SPARQL 1.1, and the Graph Store
Protocol together define a complete model for working with named
graphs. Introducing a Python-level dispatch layer AND an ontological
overlay for graph types means two new things that the standards
already cover. Users who know the RDF stack have to learn additional
conventions.

The RDF community has spent decades pushing back against proprietary
abstractions. A library that ships its own dispatch protocol and its
own ontology for things-about-graphs looks like that pattern. It
signals "this is not a general-purpose tool; it's a framework with
opinions."

**Preliminary counter:** The standards define transport (GSP) and
query (SPARQL), not application-level semantics. The four-graph
holon model is an application pattern on top of RDF, not a
replacement for it. The typed-graph ontology uses RDFS in the way
RDFS is meant to be used: to declare application-specific classes.
The protocol handles the things GSP and SPARQL leave
implementation-defined (default graph, SHACL dispatch, bulk load,
graph metadata operations).

The test: can a user who knows only SPARQL and GSP query a
holonic-populated Fuseki and get useful answers? Yes. The CGA
ontology uses standard RDF triples; the type declarations are
standard RDFS. Nothing in the data model is proprietary. The
protocol is Python convenience, not a wire format.

The counter is weakest here: the line between "convenience
abstraction" and "proprietary framework" is in the eye of the
beholder. Publishing the CGA ontology openly, documenting the
library's SPARQL queries so users can reproduce them without the
library, and being explicit that the library is a dispatch layer
not a substrate would help address this.

> **Note:** Consider explicitly stating in the README and ontology
> that the CGA ontology is open and the library is a dispatch layer,
> not a competing wire protocol.

---

### 1.2 "Typed graphs conflict with RDF 1.1's dataset model."

**The argument:** RDF 1.1 defines a dataset as a default graph plus
zero or more named graphs. A named graph is a pair of an IRI and a
graph of triples. The spec does not define semantics for "the IRI is
also a resource that can have triples about it." Some stores support
this (via quad stores treating graph names as subjects), but it's not
universal.

Making graphs into ontological resources with RDF types risks
depending on behavior that's implementation-defined. A backend that
strictly implements RDF 1.1 without extensions may not let you write
`<graph-iri> a cga:InteriorGraph` in the registry and have it
behave as expected.

**Preliminary counter:** The typing triples go in the registry
graph, not in the graph being typed. The triple
`<urn:holon:foo/interior> a cga:InteriorGraph` lives inside the
registry graph. The subject IRI happens to be the same IRI that
names another graph, but RDF 1.1 has no problem with this; IRIs are
just IRIs. The registry graph says "the IRI
`<urn:holon:foo/interior>` denotes a resource of type
`cga:InteriorGraph`." It does not claim anything about the
default-graph behavior of that IRI.

Every compliant RDF 1.1 dataset can store this pattern. The reader
does a join: "find all IRIs typed as InteriorGraph" yields graph
IRIs. Then `GRAPH <that-iri> { ... }` accesses the content.

The counter is weakest here: some RDF tooling is confused by the
pattern where a graph IRI is also a subject. Such tooling may assume
the triple describes the content, not the container. This should be
documented clearly and example queries should separate container-
metadata (in the registry) from content (in the named graph itself).

> **Note:** Identify specific RDF tools that may be confused by this
> pattern (TopBraid, Protege, PoolParty, etc.) and test against them
> if interoperability with those tools matters.

### 1.3 "Graph derivation should use PROV-O, not a custom property."

**The argument:** PROV-O already has `prov:wasDerivedFrom` for
entity-to-entity derivation between entities generated by some
activity. Inventing a CGA-specific derivation property for graph-
to-graph relationships creates a parallel vocabulary that confuses
provenance tooling and RDF-literate readers.

**Preliminary counter:** The holonic library already defines
`cga:derivedFrom` and the 0.3.1 ontology states it explicitly:
`cga:derivedFrom` and `prov:wasDerivedFrom` model distinct concepts
and coexist, neither replacing the other.

- `prov:wasDerivedFrom` is event-level derivation. A portal
  traversal's output interior was derived-from the input interior
  via a specific `prov:Activity`. This is precisely what PROV-O was
  designed for, and the library uses it that way for graph-to-graph
  derivation produced by traversals.
- `cga:derivedFrom` is persistent holon-to-holon structural
  dependency. It declares "this holon's identity continues to depend
  on that holon" without requiring an activity. It is weaker than
  PROV-O derivation: it makes no claim about when the dependency was
  established or by whom.

Both predicates are needed because they answer different questions.
"Where did this specific graph's triples come from?" uses
`prov:wasDerivedFrom`. "What does this holon structurally depend
on?" uses `cga:derivedFrom`.

The ontology's rdfs:seeAlso between them makes the relationship
explicit. The SPEC's R5.2 and R5.3 make it a normative requirement.

The counter is weakest here: the distinction is subtle. The default
assumption of a PROV-O-literate reader is that all derivation
predicates are redundant with `prov:wasDerivedFrom`. The ontology
definition and the SPEC need to make the distinction visible without
forcing the reader to infer it.

> **Note:** The clarified ontology definition (0.3.2) includes a
> rdfs:seeAlso linking `cga:derivedFrom` to `prov:wasDerivedFrom`
> and a full skos:definition explaining the split. Verify that
> downstream SHACL shapes that constrain derivation treat the two
> as distinct and do not require one to imply the other.

## 2. Implementation complexity objections

### 2.1 "The protocol is already too big. Typed graphs make it bigger."

**The argument:** Issue #4 noted the protocol has 15+ methods.
Adding graph-metadata operations (`get_graph_metadata`,
`set_graph_type`, `update_graph_stats`, scope resolution) pushes it
toward 25+. Every new backend implementation has to cover all of
them. Every new method is a potential source of backend-specific
bugs.

**Preliminary counter:** The protocol should distinguish mandatory
from optional methods. Mandatory: GSP+SPARQL operations (query,
construct, ask, update, graph CRUD). Optional, with default
implementations in a mixin: graph metadata, scope resolution, bulk
load, text/vector search. A backend that implements only the
mandatory surface still works; the optional methods get sensible
defaults (e.g. scope resolution via sequential SPARQL queries, bulk
load via chunked INSERT DATA).

The default implementations use only the mandatory methods, so they
work against any backend. Backends with native support for an
optional method override it for performance.

The counter is weakest here: Python's `Protocol` mechanism doesn't
cleanly support "mandatory vs optional with defaults." The
implementation would need either an abstract base class with
`@abstractmethod` + concrete methods, or two protocols. Both add
indirection. Tolerable, but not free.

> **Note:** Prototype the mandatory/optional split before committing
> to it. The ergonomics of the two-protocol approach vs the ABC
> approach differ in ways that matter for backend implementors.
> This is SPEC.md R9.4, scheduled for 0.4.0.

---

### 2.2 "Materialized class inventories drift from ground truth."

**The argument:** `SPEC.md` R9.2 proposes a materialized class
inventory per holon with TTL-based refresh. Materialized views are a
known source of bugs. They get out of sync under: direct SPARQL
UPDATE bypassing the library, external writes by other systems,
failed or partial refresh, concurrent writes where refresh happens
before all writes complete.

Any query against the materialized inventory may return wrong
answers. This is a correctness problem, not just performance.

**Preliminary counter:** Three mitigations:

1. Treat the inventory as a HINT, not ground truth. Every consumer
   should tolerate staleness. The console says "last refreshed N
   seconds ago" next to inventory-derived displays. Discovery
   operations use the inventory to narrow candidates, then verify
   against the live graph before returning.

2. Invalidate eagerly on known writes. Any path through the library
   that writes to a holon's interior marks the inventory stale.
   Reads from `holon_interior_classes` check the staleness flag and
   refresh if needed.

3. Offer an "authoritative" mode that skips the materialized
   inventory and queries the live graph. Slower but correct.

The counter is weakest here: "treat as a hint" requires every
consumer to know the inventory is a hint. If a downstream project
assumes it's authoritative, it will produce broken results. The
field name should signal hint-ness (`cached_classes`?
`inventory_hint`?) and the staleness model should be documented
prominently.

> **Note:** Name the inventory field to signal non-authoritativeness.
> "class_inventory" is ambiguous; "cached_interior_classes" or
> "class_inventory_hint" are clearer about the semantics.

---

### 2.3 "The registry becomes a hotspot."

**The argument:** With ontological graph categories, every operation
involves a registry query. Create a holon: write registry entries for
the holon and its four graph types. Traverse a portal: read registry
for source/target, write registry for class inventory, write registry
for derivation. Discover holons: read registry for candidates.

The registry is one named graph. Every holonic operation touches it.
In a multi-writer deployment, the registry becomes the serialization
point.

**Preliminary counter:** The registry is metadata, not data. Metadata
writes are inherently less frequent than data writes. A portal
traversal writes N data triples and 2-3 registry triples. The data
write dominates.

For discovery, the registry is read-mostly. Read concurrency is not
a throughput problem for any modern triple store.

For high-concurrency writes, the registry could be sharded by holon
(each holon's metadata in its own named graph, with a small index in
the root registry). This is a future-work mitigation.

The counter is weakest here: the sharded-registry design has not been
worked out. If concurrent writes become a bottleneck before that
design exists, the fix will need to be retrofitted under pressure.
Prototyping multi-writer scenarios early and measuring would reduce
this risk.

> **Note:** Consider creating a benchmark that measures registry
> contention under concurrent portal traversals before the typed-
> graphs work ships in 0.3.3. If the benchmark shows no contention
> at expected concurrency levels, document the result. If it shows
> contention, design the shard before shipping.

## 3. Performance objections

### 3.1 "Scope resolution adds latency to every discovery query."

**The argument:** A flat `list_holons(filter="has class X")` is one
SPARQL query. With scope resolution, the same query triggers (a) a
registry read for the starting holon's neighborhood, (b) per-level
filtering using the class inventory, (c) per-candidate verification
against the live graph. Many queries where one used to suffice.

**Preliminary counter:** Scope resolution is a separate operation
from flat listing. The library offers both:
- `list_holons_summary(filter)` — one registry query, flat results
- `resolve(predicate, from_holon, max_depth)` — scoped walk,
  priority-ordered results

The console picks the right one per view. The browser's filter bar
uses `list_holons_summary` (fast). The neighborhood view uses
`holon_neighborhood` (bounded BFS). The forthcoming `resolve`
handles scoped predicate matching. Users don't pay scope-resolution
cost unless the operation needs it.

The counter is weakest here: not all discovery queries fit cleanly
into one shape. "Find all holons with class X within 3 hops of
holon Y" is a hybrid. The library needs a composable API that lets
callers combine structural scope with attribute filtering without
paying for the full resolution walk when they don't need ordering.

---

### 3.2 "Graph-level metadata updates thrash the registry."

**The argument:** If every portal traversal updates timestamps,
triple counts, and class inventories, the registry is written on
every data operation. For high-throughput pipelines, registry write
amplification dominates.

**Preliminary counter:** The library should offer tunable
granularity:

- **Eager** (default for interactive use): update on every traversal
- **Batched**: accumulate deltas in memory, flush on timer or session
  end
- **Lazy**: update only when explicitly requested

The mode is chosen per-HolonicDataset at construction or per-
operation via keyword argument. See SPEC.md Open Question 5.

The counter is weakest here: "tunable" requires the user to
understand the tradeoffs. Sensible defaults for common cases (eager
for interactive, lazy for batch pipelines) should be the starting
point, with clear documentation of what each mode costs.

---

### 3.3 "SHACL over HTTP is slow."

**The argument:** Dispatching SHACL validation to the backend via
HTTP adds 100-500ms per round-trip compared to in-process pyshacl.
In interactive consoles and automated pipelines, this compounds.

**Preliminary counter:** Three strategies:
1. Cache reports by (data graph hash, shapes graph hash).
2. In-process fallback via pyshacl when graph sizes permit.
3. Async validation where correctness allows: portal traversal
   proceeds optimistically; the validation report flags COMPROMISED
   results retrospectively rather than blocking.

The counter is weakest here: async validation changes membrane
semantics. Some deployments (defense C2, safety-critical) will
reject the optimistic model and require synchronous blocking. The
library needs to make explicit which semantics the caller gets.

> **Note:** The sync-vs-async SHACL question is a real semantic
> fork, not just a performance tuning knob. If async is offered, it
> should be opt-in with clear naming (e.g.
> `traverse_portal_optimistic`) and the default should remain
> synchronous blocking.

## 4. Semantic and ontological objections

### 4.1 "Graph categories conflate content and container."

**The argument:** An `InteriorGraph` is a container. Typing the
container as `cga:InteriorGraph` conflates the container with the
content. The content IS the holon's interior data; the container is
just a named region of the dataset. Typing the region is like typing
the filename rather than the file.

**Preliminary counter:** The confusion is resolved by the precise
meaning of `rdf:type`. `<graph-iri> a cga:InteriorGraph` means "the
resource named by this IRI is of type InteriorGraph." The resource
IS the container. The container is the structural role the named
graph plays in the holon. The content is separate and not typed by
this triple.

This parallels how files are typed: `file.txt a text/plain`
describes the container (the file), not the content (the words).

The counter is weakest here: RDF's `rdf:type` doesn't usually type
containers in this way. Most RDF typing is about the denoted
resource's properties, not about its role in a structure. The
convention should be documented explicitly in the ontology's
`rdfs:comment`.

---

### 4.2 "Graph categories should be roles, not types."

**The argument:** The same named graph might play different roles in
different contexts. A provenance-aggregator holon's interior is both
an interior (from the holon's perspective) and a provenance store
(from the consumer's perspective). Typing it as
`cga:InteriorGraph, cga:ProvenanceGraph` forces both perspectives
into a type hierarchy, which breaks if the two perspectives become
incompatible later.

Roles, not types: `<graph-iri> cga:graphRole cga:InteriorRole`.
Roles are first-class and can coexist; types should be for intrinsic
identity.

**Preliminary counter:** SPEC.md Open Question 2 flags this as
unresolved and recommends adopting flat+roles because the
dual-role case is real (`cga:AggregateHolon`, provenance aggregators,
index-plus-interior patterns). A flat hierarchy with roles (single
`cga:HolonicGraph` type, multiple `cga:graphRole` values) has the
properties the critic wants. The cost is that the common case is
`?g cga:graphRole cga:InteriorRole` instead of `?g a
cga:InteriorGraph`. One more triple pattern per query.

The CGA ontology already uses this pattern internally for
`cga:LayerRole` individuals (`cga:InteriorRole`, `cga:BoundaryRole`,
etc.), so extending it to graphs is consistent with existing
convention.

The counter is weakest here: the evidence for how common the
multi-role case is in practice is limited. If most graphs have a
single role, types would be slightly more ergonomic. The recommended
approach is to ship flat+roles in 0.3.2 (additive, matches the
existing LayerRole pattern) and revisit if the evidence warrants a
type-based layer.

> **Note:** Prototype both approaches against the holonic-console
> NIGHTWING fixture and compare query ergonomics before committing.

---

### 4.3 "The CGA ontology is not open enough."

**The argument:** The CGA ontology defines application-specific
classes. Other projects using named graphs in holonic-like patterns
will have their own ontologies. If interoperability matters
(federation, cross-organization alignment), the categories need to
be expressible in a more neutral vocabulary.

**Preliminary counter:** The CGA ontology should be aligned with
neutral vocabularies where they exist:
- SKOS for category hierarchies
- DCAT for dataset-level metadata
- PROV-O for derivation and provenance (already done)
- VoID for graph statistics

Each alignment is a single triple. Users who prefer neutral
vocabulary get interoperability; users who prefer CGA keep shorter
queries.

The counter is weakest here: alignments tend to rot. If DCAT or VoID
evolves, alignment triples need updating. Not all alignments are
clean 1:1 mappings. Selectivity about which alignments to publish
and periodic review are necessary.

> **Note:** PROV-O alignment is already adopted (R5, R6.7, objection
> 1.3). DCAT and VoID alignment for graph-level metadata would be
> natural additions in 0.3.3 alongside the metadata work. SKOS
> alignment is lower priority unless cross-organization federation
> is near-term.

## 5. Federation and interoperability objections

### 5.1 "Registry-to-registry federation is speculative."

**The argument:** SPEC.md mentions that typed registries enable
federated deployment via `cga:federatesWith`. No production
deployment needs this today, and designing for it risks
over-engineering.

**Preliminary counter:** Federation does not need to be implemented
now. It only needs to not be closed off. Ontological typing and
graph-level metadata are useful in a single-registry deployment
(discoverability, extensibility); federation is a bonus they make
possible. If federation never arrives, nothing extra was paid. If
it does, the hooks exist.

The counter-risk: designing for a speculative use case may lead to
API shapes that don't fit any actual use case well. Mitigation: the
typing and metadata are adopted because they pay off for single-
registry use. Federation is mentioned as a consequence, not a
requirement.

The counter is weakest here: the discipline to not over-engineer for
federation is hard to maintain during implementation. "Should method
X work across federated registries?" will arise during design. The
answer should be "not yet, but don't make it impossible."

---

### 5.2 "Other RDF libraries don't work this way."

**The argument:** rdflib, SPARQLWrapper, pyoxigraph don't have a
dispatch protocol or graph-type ontology. Users coming from those
libraries have to learn additional conventions. The patterns don't
generalize back to the wider ecosystem.

**Preliminary counter:** Those libraries are graph-access
foundations. They don't prescribe application architecture because
they don't have one. Holonic is an application architecture. It
prescribes patterns because those patterns are the value
proposition. The two layers are not in competition.

The counter is weakest here: the boundary between "foundation" and
"application" is fuzzy. The library should be explicit in the README
about what it IS (holonic architecture implementation) and is NOT
(a general-purpose RDF toolkit).

---

## 6. Migration and compatibility objections

### 6.1 "Existing holonic deployments break."

**The argument:** Any deployment with existing registry graphs
without typed-graph triples needs a migration. If not automated,
deployments will skip it and run with mixed typed/untyped
registries, producing partial results.

**Preliminary counter:** The library detects the absence of typed
graphs at startup and offers a one-shot idempotent migration: for
every holon, add typing triples for its four canonical graphs.
Reversible by removing the typing triples.

Queries written for typed graphs fall back gracefully: `?g a
cga:InteriorGraph` produces no results, then the library retries
with `?h cga:hasInterior ?g`. Old deployments keep working until
the operator runs the migration.

The counter is weakest here: automated migrations that run at
startup are a known production-incident source. The migration should
be an explicit CLI command (`holonic-migrate-registry`) run during a
maintenance window, not an automatic startup behavior.

> **Note:** Ship the migration as a CLI tool, not as startup
> behavior. Document the migration steps and the rollback procedure
> in the 0.3.2 or 0.3.3 release notes.

---

### 6.2 "Rename churn."

**The argument:** Issue #4 Option D includes renaming `GraphBackend`.
Every downstream project that imports it has to update.
Documentation, examples, tutorials all need updating.

**Preliminary counter:** Keep `GraphBackend` as a deprecated alias
for the entire 0.4.x series after the rename. Downstream projects
see a deprecation warning and switch on their own schedule. Remove
the alias only in 0.5.0 with advance notice.

The counter is weakest here: deprecated aliases accumulate cruft if
left too long. The specific removal version should be committed to
and communicated clearly.

## 7. User mental-model objections

### 7.1 "Users think in objects, not triples."

**The argument:** Python developers default to object-oriented
thinking. A `Holon` class with `.interior`, `.boundary`, `.context`
attributes matches their mental model. Dataclasses returned by
queries, with the developer running the right query at the right
time, does not.

By going graph-native, the library forces users to think in RDF.
This is a learning curve and a maintenance burden.

**Preliminary counter:** The graph-native rewrite (v0.2.1) was
deliberate. Python objects that cache RDF state have known problems
(staleness, invalidation). The graph IS the data model.

The API can provide convenience accessors that look object-like
while being query-backed. Every access is a fresh query. Slower than
cached access but always correct. The `console_model` dataclasses
in 0.3.1 follow this pattern: they are return types from specific
queries, not persistent domain objects.

The counter is weakest here: "every access is a query" can surprise
users who assume repeated access is cheap. The semantics should be
documented clearly. An explicit per-request cache the caller opts
into could address performance-sensitive use cases.

---

### 7.2 "Typed graphs add cognitive load without visible benefit."

**The argument:** For a user who just wants to create a holon and
traverse a portal, the graph categories are invisible overhead. They
learn about `cga:InteriorGraph` when inspecting the registry and
seeing unfamiliar triples. Every user pays the cognitive cost; only
advanced users (discovery, federation) see the benefit.

**Preliminary counter:** The typing triples are added by the library
automatically when registering a holon. Users working at the
`HolonicDataset` API level never see them. The triples appear in
queries only when the user writes a query that touches them.

For users inspecting their graph store directly (Yasgui, Fuseki
admin), the triples will be visible. The tutorial and examples
should introduce the typed-graph pattern early.

---

### 7.3 "The library is prescriptive and we want descriptive."

**The argument:** Some teams have existing RDF data with their own
conventions. They want a library that helps them work with RDF, not
one that insists their data conform to a specific pattern.
Prescriptive libraries have smaller addressable markets.

**Preliminary counter:** Prescriptive is the right tradeoff for
this library. The value proposition IS the holonic pattern. Teams
that don't want the pattern should use rdflib directly. The library
exists because the pattern is valuable and implementing it correctly
every time is tedious.

The counter is weakest here: the audience for "teams that want the
holonic pattern" is small today. Growing it requires writing,
speaking, and evangelism. If broader adoption is a goal, the path is
teaching why the pattern matters (easier but slower), not relaxing
the prescriptivism (harder to do well and risks diluting the value).

> **Note:** Consider whether the prescriptivism boundary is drawn in
> the right place. The four-graph model, portals, and membranes are
> the valuable prescriptive core. The IRI naming convention, the
> ontology namespace, and the metadata conventions are more
> negotiable.

---

## 8. Meta-objection: the sunk-cost check

**The argument:** The library has an existing architecture. Arguments
for changes are biased by the existing investment. An unbiased
observer starting from scratch might not choose this architecture.
Periodically asking "if we started over, would we build this?" is a
useful discipline.

**Preliminary assessment:**
- Graph-native over object-native: keep.
- Four-graph holon model with explicit LayerRole individuals: keep.
- `GraphBackend` protocol: keep, but smaller. Mandatory core +
  optional mixin (SPEC.md R9.4, 0.4.0).
- CGA ontology as application layer: keep.
- Python as host language: keep, but with lightweight async support
  from the start (not retrofitted).
- Starting from rdflib as default backend: probably keep. A lighter
  in-process option (dict-backed triple store) might have been
  better for the 80% case.
- Specl-style spec-driven development from the start: adopt
  retroactively (done in 0.3.1 via the pixi `spec-*` tasks).

> **Note:** Substitute your own honest assessment here. This
> exercise is worth repeating at each major version milestone.

---

## 9. How to use this document

When proposing a change to the library architecture, check it
against this list. If an objection applies to the change, decide
whether:
1. The change accepts the objection (add it to "known limits")
2. The change mitigates the objection (document how)
3. The change invalidates the objection (document why)

If a new objection surfaces that is not in this list, add it. The
document grows over time as blind spots are discovered.
